## Project Overview

This is the official Terraform provider for ConductorOne, an identity access management platform. The provider is **generated using Speakeasy** from an OpenAPI specification, meaning most code is auto-generated and should not be manually edited except in designated areas.

## Build & Test Commands

```bash
make build         # Build the provider
make test          # Run unit tests (go test -v -cover -timeout=120s -parallel=4 ./...)
make testacc       # Run acceptance tests (requires TF_ACC=1, timeout 5m)
make lint          # Run golangci-lint
make fmt           # Format code with gofmt
make gen           # Regenerate SDK from OpenAPI spec using Speakeasy (includes vendor)
make vendor        # Run go mod tidy && go mod vendor
make pre-commit    # Run all validation checks (build, lint, test, generate)
make install-hooks # Install git pre-push hook for automatic validation
```

**Local testing workflow:**
```bash
go run main.go --debug
# Copy the TF_REATTACH_PROVIDERS env var output
# In another terminal:
cd examples/<your-example>
TF_REATTACH_PROVIDERS=... terraform init
TF_REATTACH_PROVIDERS=... terraform apply
```

## Architecture

### Directory Structure

- `internal/provider/` - Terraform provider resources and data sources
- `internal/sdk/` - Auto-generated ConductorOne API client (from Speakeasy)
- `internal/planmodifiers/` - Custom Terraform plan modifiers by type
- `internal/validators/` - Custom Terraform validators (date, JSON, RFC3339, etc.)

### Code Generation

Most code is generated by Speakeasy from the OpenAPI spec:
- `openapi.yaml` / `openapi-applied.yaml` - API specification
- `overlay.yaml` - Speakeasy overlays for customization
- `gen.yaml` - Speakeasy generation configuration

**Two generation flows:**
- `make gen` - Downloads fresh spec from `https://insulator.conductor.one/api/v1/openapi.yaml`, applies overlays, and regenerates. Use this to sync with the latest API.
- `speakeasy run` - Regenerates from the existing local `openapi.yaml` (via `.speakeasy/workflow.yaml`) without overwriting. Use this to regenerate after manual overlay changes.

The local `openapi.yaml` reflects the API state at the time of last `make gen`, not a versioned artifact.

Files with `// Code generated by Speakeasy` headers should not be manually edited.

### Speakeasy Version Pin

**IMPORTANT:** Speakeasy is pinned to version 1.658.1 in `.speakeasy/workflow.yaml`. As of December 2025, newer versions introduce a regression causing perpetual diffs in Terraform state (fields like `provision_policy` reset on every refresh). Do not upgrade until the regression is fixed upstream.

### Pagination Fix Verification

After any `make gen` or `speakeasy run`, verify the pagination fix is intact. Older Speakeasy versions had a bug where pagination loops used variable shadowing instead of assignment, breaking multi-page result iteration.

**Correct pattern** (must be present in data sources like `apps_data_source.go`):
```go
for {
    var err error
    res, err = res.Next()   // Assignment (=), reuses outer res
    if err != nil { ... }
    if res == nil { break }
    // process res
}
```

**Broken pattern** (if you see this, the pagination fix regressed):
```go
for {
    res, err := res.Next()  // Declaration (:=), shadows outer res - BROKEN
    ...
}
```

Quick verification command:
```bash
grep -A2 "for {" internal/provider/apps_data_source.go | grep -E "res, err =|var err"
```

Should output lines with `var err error` and `res, err = res.Next()` (assignment, not declaration).

### Validation Workflow

After running `make gen` or making changes, validate before committing:

```bash
make pre-commit
```

This runs build, lint, test, and generate (tfplugindocs). Review any changes from `make generate` and include them in your commit.

**Git hooks (recommended):** Install the pre-push hook to automatically validate before pushing:

```bash
make install-hooks   # Run once after cloning
```

This configures git to run `make pre-commit` before each push. Use `git push --no-verify` to bypass in emergencies.

### Provider Structure

The provider (`internal/provider/provider.go`) exposes:
- **Resources**: Core ConductorOne entities (apps, policies, webhooks, access profiles, etc.)
- **Integration Resources**: 140+ connector integrations (AWS, Okta, GitHub, etc.) in `integrations.go`
- **Data Sources**: Read-only lookups for existing resources

Each resource typically has three files:
- `*_resource.go` - Terraform resource implementation (schema, CRUD operations)
- `*_resource_sdk.go` - SDK conversion helpers (Terraform ↔ API models)
- `*_data_source.go` / `*_data_source_sdk.go` - Data source equivalents

### SDK Client

The `internal/sdk/` package provides the generated API client:
- `conductoroneapi.go` - Main API client with service accessors
- `auth.go`, `token_source.go` - OAuth2 client credentials authentication
- `models/` - Request/response types

### Authentication

Provider accepts credentials via:
- Provider config: `client_id`, `client_secret`, `tenant_domain`
- Environment: `CONDUCTORONE_CLIENT_ID`, `CONDUCTORONE_CLIENT_SECRET`, `CONDUCTORONE_TENANT_DOMAIN`

## Key Patterns

- Resources use HashiCorp's terraform-plugin-framework (not the older SDK)
- Plan modifiers handle computed fields and state management
- Validators provide custom validation logic for ConductorOne-specific types
- Integration resources follow a consistent pattern with per-connector configuration schemas
